# NexLink 荣誉与声誉（Honor & Reputation）

> **状态：合约机制已可用（SDK-ready）；荣誉墙 UX 与发行流程处于设计 / 提案阶段。** 灵魂绑定代币（荣誉的链上载体）**现在即可部署与铸造**（见 [NFT.md](NFT.md)）。官方**荣誉墙**、面向组织的发行流程、以及不良记录策略均为提案。本文档是 dApp 对接的产品层概念；[NFT.md](NFT.md) 是合约层参考。

**荣誉（荣誉）**是组织颁发给个人的**灵魂代币（SBT，灵魂代币）**，用于证明其**无法购买、确实获得**的东西——学历、赛事名次、科研或工作成果——或一条**不良记录**（欠债、违规）。一个人的荣誉合起来构成其**信任体系**，并以应用内的**荣誉墙**呈现。

---

## 1. 荣誉是什么

| 属性 | 说明 |
|---|---|
| **不可转让** | 荣誉是 SBT——**不能出售、发送或转移**（[NFT.md 第 4 节](NFT.md#4-soulbound-tokens-sbt)）。这正是它可信的原因：它证明持有者确实获得过。 |
| **仅发行方可收回** | 只有**发行方**能收回（销毁）它——例如债务已偿还 → 发行方清除该不良记录。 |
| **绑定到身份、汇总到主身份** | 荣誉绑定到获得它的身份，且**一个人的所有荣誉都会汇总到其主身份**——见 [IDENTITY.md 第 3 节](IDENTITY.md)。 |

| 类别（原文） | 示例 | 极性 |
|---|---|---|
| 学历证书 | 毕业证、资格认证 | 正面 |
| 赛事名次与奖杯 | 比赛名次、奖杯 | 正面 |
| 科研成果 / 工作成果 | 科研产出、工作成就 | 正面 |
| **不良记录** | 不还信用卡、工作违规 | **负面** |

这些正是经典的**灵魂绑定代币**应用场景（身份验证、链上信用、DAO 防女巫、凭证）在 NexLink 中的落地。

## 2. 发行一枚荣誉（作为 dApp / 组织）

一枚荣誉即一次灵魂绑定 ERC-721 铸造。**须先向基金会申请根证书**（第 4 节）——没有根 SBT，你的荣誉不会被计入；随后从你已认证的合约铸造。合约机制见 [NFT.md](NFT.md)；荣誉特有的规则如下：

```javascript
// 向接收方身份的地址铸造一枚灵魂绑定荣誉。
//（使用灵魂绑定合约——转让会回滚；见 NFT.md 第 4 节。）
const { txHash } = await NexlinkApp.contract.call({
  contract: HONOR_COLLECTION_ADDRESS,
  abi: SOULBOUND_NFT_ABI,
  method: "mint",
  args: [recipientAddress, metadataURI]   // 接收方 = 获得该荣誉的身份地址
});
```

**荣誉元数据**在标准 ERC-721 结构（[NFT.md 第 6 节](NFT.md#6-metadata)）之上扩展声誉字段：

```json
{
  "name": "计算机科学硕士",
  "image": "ipfs://…",
  "attributes": [
    { "trait_type": "category", "value": "degree|trophy|research|work|negative" },
    { "trait_type": "polarity", "value": "positive|negative" },
    { "trait_type": "issuer",   "value": "清华大学" },
    { "trait_type": "issued_at","value": "2025-06-30" },
    { "trait_type": "evidence", "value": "ipfs://…" }
  ]
}
```

请将元数据托管在 IPFS 上，使其在铸造后不可篡改。

## 3. 不良记录——完整性规则

不良记录是 `polarity: negative` 的荣誉。三条规则让它有意义：

| 规则 | 原因 |
|---|---|
| **不可转让** | 不能转移出去以逃避 |
| **持有者不可自行销毁** | 持有者不得删除自己的不良记录——**只有发行方**可收回 |
| **汇总到主身份** | 在每人仅一个的主身份处显现，因此**无法通过切换身份来规避**（[IDENTITY.md 第 3 节](IDENTITY.md)） |

> [NFT.md 第 4 节](NFT.md#4-soulbound-tokens-sbt) 的通用灵魂绑定合约允许持有者销毁。**不良记录合约必须移除持有者销毁能力**，并将收回限定给发行方。

## 4. 信任锚定——基金会是根 CA

荣誉通过链上**证书层级**（去中心化 PKI）来把关，防止市场充斥假凭证。

- **NexLink 基金会是根 CA。** 它维护唯一的[身份合约](IDENTITY.md)，并锚定所有荣誉信任。
- **任何人都可以运行自己的荣誉合约，但必须先向基金会申请根证书（根证书 / 根 SBT / 通行证）。** 你部署荣誉合约后，基金会**核验并发放根证书**——即一枚由你的发行方钱包持有的根 SBT。
- 根证书有**三种类型**，每种都是一类根 SBT：

| 根证书类型 | 原文 | 颁发对象 |
|---|---|---|
| **政府** | 政府 | 政府机构 |
| **基金会** | 基金会 | 基金会本身 |
| **其他机构** | 其他机构 | 其他受认可的组织 |

- 只有持有根 SBT 者才能从其已核验的合约铸造荣誉 SBT。荣誉的可信度来自**其发行方持有的根证书类型**——荣誉墙按等级为荣誉加徽章，**从发行方的根 SBT 链上读取**（绝不采信可伪造的元数据）。

```
NexLink 基金会（根 CA）
  └── 根 SBT（通行证）—— 三种类型：政府 / 基金会 / 其他机构
        └── 你已核验的荣誉合约 → 荣誉 SBT → 身份的钱包
```

## 5. 不暴露地证明荣誉（ZK）

dApp 常常需要知道某人**达到某个门槛**（有信用、无不良记录、持有某凭证），而无需看到其完整履历——尤其当对方以**匿名身份**活动时。这是针对该人汇总荣誉、由**主身份**签发的**零知识证明**：

- dApp 设定**约束条件（约束条件）**；用户的主身份以 ZK 方式证明；身份从不泄露。
- 流程与拟定的 `NexlinkApp.identity.prove()` 接口见 [IDENTITY.md 第 4 节](IDENTITY.md)。
- **担保（[ESCROW.md](ESCROW.md)）是旗舰消费方**——为一笔担保交易设定 ZK 信用 / 声誉门槛。

## 6. 展示荣誉

- **两处展示**——身份 SBT 与荣誉 SBT 同时在**钱包**（如同任何持有的代币）与**个人主页**的荣誉墙中显示。
- **荣誉墙**——正面与负面分区、按类别分组，每枚荣誉按其发行方的**根证书等级**（政府/基金会/其他机构）加徽章。
- **单身份视图 vs 汇总视图**——某个人格展示其自身的荣誉；**主身份**展示**汇总后**的人层面集合。
- **在 dApp 中**——读取某合约对某接收方的 `balanceOf` / `tokenURI`，或（推荐）查询后端荣誉索引器，而非逐条链上枚举。
